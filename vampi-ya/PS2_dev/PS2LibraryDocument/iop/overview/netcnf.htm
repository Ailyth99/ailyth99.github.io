1 概要

netcnf.irxは、「あなたのネットワーク設定ファイル」(共通ネットワーク設定ファイル)を操作するための機能を提供する常駐ライブラリです。
共有リソースである「あなたのネットワーク設定ファイル」を扱うネットワーク設定アプリケーションを作成するためには共通ネットワーク設定ライブラリ(netcnf.irx) を使用し、また作成基準を守らなければなりません。作成基準は、EE/IOP共通文書ディレクトリ(genera/)の以下の資料を参照してください。

  • あなたのネットワーク設定ファイル使用規定(yncf)

2 NET設定ファイルの概要

ネットワーク内でのIPアドレスやダイヤルアップ接続のためのアクセスポイントの電話番号など、ネットワーク接続に必要な個別情報は、このドキュメントで定める仕様にしたがった設定ファイルとして、個々の"Playstation 2"ごとに保存します。設定ファイルを生成 / 編集 / 解釈するすべてのプログラムはこの仕様に従わなければなりません。

2. 1 NET設定ファイルの種別

NET設定ファイルには以下の3種があります。設定ファイル中に記述できるキーワードは、設定ファイルの種別ごとに定義されています。

 ファイル種別   内容 
 NET_CNF   回線に関する情報、他の設定ファイルのファイル名 
 ATTACH_CNF   あるひとつのネットワークインタフェースに関する設定情報、 
 つまり、ネットワークデバイスが接続されたときに行う 
 処理の指定。 
 DIAL_CNF   あるひとつのモデムの発信手続き情報 

2. 2 未定義のキーワード

将来新しいキーワードが追加される場合に備えて、NET設定ファイルを編集するプログラムは、元のファイル中に未定義または未対応のキーワードが現れた場合は、変更を加えずそのまま保存するようにしなければなりません。

2. 3 設定処理

NET設定ファイルの内容を解釈して設定を行う処理は、netcnf.irxを起動することによって行います。詳細はnetcnf.irxのドキュメントを参照してください。

2. 4 netcnf.irxの起動方法

netcnf.irx は以下の方法でロードして下さい。
netcnf.irx [<option>] icon=<icon-path> iconsys=<iconsys-path>

注意：以下の引数を必ずつけるようにしてください。
 icon   アイコンファイル 
 iconsys   アイコン管理用ファイル(icon.sys) 

オプションには以下のものが用意されています。

  • -no_check_capacity
    sceNetCnfAddEntry(), sceNetCnfEditEntry()関数にて容量チェックを行いません。
  • -no_check_provider
    sceNetCnfAddEntry(),sceNetCnfEditEntry()関数にて特殊プロバイダのチェックを行いません。

3 NET設定ファイルの文法

NET設定ファイルを記述する際の基本的な規則について説明します。

3. 1 一般的な形式

NET設定ファイルは「\n」（LF:0x0a）またはファイルの終わりを終端とする「行」を単位として解釈されます。ただし「\n」の直前に「\\」（0x5c）を置くと、その「\\」と「\n」の2bytesは無視され、次の行は論理的に継続しているものとみなされます。

一般にNET設定ファイル中の行は次の形式を持ちます。

keyword  arg1  arg2 ...

キーワードとその引数は、ひとつの論理行にただ一組だけ記述することができます。キーワードと引数それぞれは、1個以上の連続するスペース（SP:0x20）またはタブ（TAB:0x08）によって区切ります。スペースとタブ以外にも、IOPの開発環境中のctype.hで定義されているisspace()マクロによって空白とみなされる文字は区切り文字となります。

3. 2 コメント

行の先頭に「#」を置くと、その行はコメント行とみなされます。
また、文字列中以外の箇所に「#」を置くと、その行の終わりまでの部分はコメントとみなされます。

3. 3 CRコード

NET設定ファイル中では、CRコード（0x0d）はエスケープ指定されている場合を除いて、すべて無視されます。

3. 4 文字列および文字列中のエスケープ処理

引数が文字列の場合、その文字列全体を「"」で囲んで記述してください。設定管理ファイル内の
usr_name にあたる設定名については「"」で囲む必要はありません。その他、例外的に「"」で
囲まなくても済むケースもありますが、一般的ではありませんのでその条件等に関しては記述を
省略します。
「"」で囲まれた文字列中には、制御コード等を以下のエスケープ指定を用いて記述することができます。

 エスケープ指定   対応する文字 
 \a   BEL（0x07） 
 \b   BS（0x08） 
 \t   TAB（0x09） 
 \n   LF（0x0a） 
 \v   VT（0x0b） 
 \f   FF（0x0c） 
 \r   CR（0x0d） 
 \ooo   8進数によるコード指定。oooは1桁から3桁の8進数。 
 \xxx   16進数によるコード指定。xxは1桁または2桁の16進数。 
 \X   文字Xの指定。Xは上記以外の文字（[^abtnvfr0-9xa-fA-F]） 
 で、主に「\」や「"」を記述するために使用する 


エスケープ文字は、実際に設定ファイルに記述する際には「\」をエスケープするため、「\\a」のように記述することになります。

3. 5 漢字コード

漢字コードは UTF8 を御使用下さい。また、設定名以外の設定項目(設定内容そのもの)について
漢字・仮名文字を使用することを禁止します。設定名においては平仮名・カタカナのみ使用可能と
します。また、「,」は区切りとして設定管理ファイル内で既定されていますので使用不可です。

3. 6 ヘッダコメント

NET設定ファイルの先頭には次のコメントを記載しなければなりません。

# <Sony Computer Entertainment Inc.> comment

「>」から行末までの内容は任意です。
このヘッダコメントは、netcnf.irxによる個体符号化および復号の際にエラー判定に利用します（個体符号化および復号の対象となるのはATTACH_CNFのみですが、ヘッダコメントはすべてのNET設定ファイルに必要です）。

4 NET_CNFのキーワード

NET_CNF形式のファイルはNET設定ファイル群のルートになるファイルで、ユーザごとの個別情報と、他の設定ファイルのファイル名を含んでいます。NET_CNF中に記述できるキーワードは以下のとおりです。

4. 1 interface

ネットワークインタフェースを定義するキーワード、より具体的に言えばATTACH_CNFを指定するキーワードです。NET_CNF中に複数記述することができます。

書式

interface "display_name" "ifc_cnf" ["dev_cnf"]

解説

display_nameは設定ファイル作成・編集プログラム上でユーザが各インタフェース定義を区別するための表示用名称です。特に制限はなく任意の文字列を指定できます。
ifc_cnfとdev_cnfは、そのネットワークインタフェースに関する設定情報が記述されているATTACH_CNFファイルのファイル名を指定します。ファイル名はNET_CNFの存在するディレクトリを基準とする相対パス名で指定することができます。
PPP接続の場合、ifc_cnfにはISP依存の設定情報、dev_cnfにはモデム依存の設定情報を記載します。この使い分けについて詳細は別途運用規定が設けられます。
ただし、dev_cnf ファイル中の type のみは常に接続処理でのマージ対象とはならず、ifc_cnf 中の type のみが使用されます。
あるキーワードがNET_CNF / ifc_cnf / dev_cnfの3つのファイルに重複して指定されている場合、後の指定が有効になります。つまり、NET_CNF ＜ ifc_cnf ＜ dev_cnfという優先順位で解釈されます。

4. 2 chat_additional

モデムに対する追加スクリプトを定義するキーワードです。

書式

chat_additional "additional_script"

解説

PPPのインタフェースにおいて、chat_initキーワードで定義される初期化スクリプトに続いてモデムに送られる追加スクリプトを定義するキーワードです。モデムの送信レベル調整のような、ユーザ固有の対応のために使用します。NET_CNFでの定義はインタフェースに依存しないデフォルトの追加スクリプトとなり、個々のインタフェースに依存する定義が必要な場合は、該当するATTACH_CNFに記述しておくとNET_CNFでの設定が上書きされます。
スクリプトの仕様については「チャットスクリプト仕様」を参照してください。

4. 3 redial_count

リダイヤル回数の上限を指定するキーワードです。

書式

redial_count redial_count

解説

PPPのインタフェースにおいて、モデムドライバが「通話中」を返した際に何回までリダイヤルするかを指定するキーワードです。指定しない場合のデフォルトは0回となり、リダイヤルは行われません。
NET_CNFでの指定はインタフェースに依存しないデフォルトのリダイヤル回数となり、インタフェースごとに変更したい場合は、該当するATTACH_CNFに記述しておくとNET_CNFでの設定が上書きされます。

4. 4 redial_interval

リダイヤル間隔を指定するキーワードです。

書式

redial_interval redial_interval

解説

リダイヤルの際の遅延時間を秒単位で指定するキーワードです。指定しない場合のデフォルトは10秒です。
NET_CNFでの指定はインタフェースに依存しないデフォルトのリダイヤル間隔となり、インタフェースごとに変更したい場合は、該当するATTACH_CNFに記述しておくとNET_CNFでの設定が上書きされます。

4. 5 outside_number

外線発信文字列を指定するキーワードです。

書式

outside_number "outside_number"

解説

外線発信の際に電話番号の先頭にダイヤルする外線発信文字列を指定するキーワードです。指定がない場合のデフォルトは""（空文字列）です。

4. 6 outside_delay

外線発信遅延文字列を指定するキーワードです。

書式

outside_delay "outside_delay"

解説

外線発信の際に、外線発信文字列と電話番号の間にはさむ外線発信遅延文字列を指定するキーワードです。モデムの機種と設定に依存しますが、多くのATコマンドモデムでは「,」で2秒遅延の意味になります。指定がない場合のデフォルトは""（空文字列）です。
NET_CNFでの指定はインタフェースに依存しないデフォルトの凱旋発信遅延文字列となり、インタフェースごとに変更したい場合は、該当するATTACH_CNFに記述しておくとNET_CNFでの設定が上書きされます。
なお、外線発信文字列（outside_number）が空文字列のときは外線発信遅延文字列は使用されません。

4. 7 dialing_type

回線タイプを指定するキーワードです。

書式

dialing_type [tone|pulse|any]

解説

回線のタイプを次のいずれかで指定します。

 引数   意味 
 tone   トーン回線（アナログ）　[default] 
 pulse   パルス回線（アナログ） 
 any   上記以外の回線（デジタル等） 


NET_CNFでの指定はインタフェースに依存しないデフォルトの回線種別となり、インタフェースごとに変更したい場合は、該当するATTACH_CNFに記述しておくとNET_CNFでの設定が上書きされます。

5 ATTACH_CNFのキーワード

ATTACH_CNF形式はNETDEVレイヤに対応する設定ファイルの形式です。
ATTACH_CNFに記述するキーワードは2つのグループに分けられます。ひとつは、そのATTACH_CNFをどのNETDEVレイヤモジュール・どのネットワークデバイスに対して適用するかを指定するキーワードです。このグループに属するキーワードは次のとおりです。

 キーワード   指定内容 
 type   NETDEVレイヤの種別指定（USB-Ether / PPP） 
 vendor   ネットワークデバイスのベンダー指定 
 product   ネットワークデバイスの製品名指定 
 location   ネットワークデバイスの接続先ポート指定 


もうひとつのグループは、ネットワークI/Fやネットワークデバイスに関する設定を記述するためのキーワードです。このグループには、PPP接続専用のキーワード（下表で※のもの）が多数あります。

 キーワード   指定内容 
 dhcp   DHCPを使用するかどうかを指定 
 dhcp_host_name   DHCP サーバへ通知するホスト名を指定 
 dhcp_host_name_null_terminated   DHCP サーバへ通知するホスト名に 
 末端の NULL 文字を含めるかどうかを指定 
 dhcp_release_on_stop   インタフェースがダウンした時に  
 DHCP サーバへIP アドレスの解放を 
 通知するかどうかを指定 
 address   IPアドレスを指定 
 netmask   サブネットマスクを指定 
 nameserver   ネームサーバを指定 
 route   ルーティング情報を指定 
 phy_config   物理層チップの設定方法を指定 
 mtu   want.mru, allow.mru のデフォルト値、 
 MTU 値を指定 
 pppoe   PPPoE(PPP over Ethernet) を 
 有効にするかどうかを指定 
 pppoe_service_name   PPPoE(PPP over Ethernet) の 
 サービス名を指定 
 pppoe_ac_name   PPPoE(PPP over Ethernet) の 
 アクセスコンセントレータ名を指定 
 pppoe_host_uniq_auto   PPPoE(PPP over Ethernet) の 
 クライアントの判別に用いられる

 自動生成するかどうかを指定 
 chat_additional   モデムに対する追加スクリプトを定義 
 redial_count   リダイヤル回数の上限を指定 
 redial_interval   リダイヤル間隔を指定 
 outside_delay   外線発信遅延文字列を指定 
 dialing_type   回線タイプを指定 
 phone_number   発信先電話番号を指定　※ 
 answer_mode   着信によってネットワーク接続を 
 行うかどうかを指定 
 answer_timeout   着信によってネットワーク接続を 
 行う場合のタイムアウト時間を指定 
 chat_login   ログイン手順を指定　※ 
 auth_name / auth_key /  
 peer_name / peer_key 		  認証名および認証キー　※ 
 want.auth / allow.auth   認証方式　※ 
 lcp_timeout   LCPタイムアウト時間　※ 
 ipcp_timeout   IPCPタイムアウト時間　※ 
 idle_timeout   アイドルタイムアウト時間　※ 
 connect_timeout   回線接続のタイムアウト時間　※

PPP接続用のキーワードはさらに、ISPに関する設定を行うもの（ifc.cnfに記述すべきもの）と、モデムに関する設定を行うもの（dev.cnfに記述すべきもの）に分けられます。この使い分けについては運用規定が別に設けられます。

5. 1 type

INETレイヤの下位レイヤのうち、そのATTACH_CNFが対応するものの種別を指定するキーワードです。

書式

[-]type [eth|ppp|nic]

次のいずれかを指定できます。
  • -type　　　下位レイヤの種別指定なし[default]
  • type eth　　USB-Ethernetに対応
  • type ppp　　PPP接続に対応
  • type nic　　Ethernet (Network Adaptor)に対応

5. 2 vendor

そのATACCH_CNFが対応するインタフェースデバイスのベンダ名を指定するキーワードです。

書式

[-]vendor [vendor]

次のいずれかを指定できます。
  • -vendor　　　　ベンダ指定なし[default]
  • vendor vendor　ベンダ名がvendorのデバイスにのみ対応

解説

ベンダ名はUSBのString Descriptorによる名称として定義されるものではなく、そのデバイスのドライバが指定する文字列です。

5. 3 product

そのATTACH_CNFが対応するデバイスの製品名を指定するキーワードです。

書式

[-]product [product]

  • -product　　　　　製品名指定なし[default]
  • product product 　製品名がproductのデバイスにのみ対応

解説

製品名はUSBのString Descriptorによる名称として定義されるものではなく、そのデバイスのドライバが指定する文字列です。

5. 4 location

そのATTACH_CNFが対応するデバイスの位置情報、つまりデバイスが接続されているポートを指定するキーワードです。

書式

[-]location [location]

次のいずれかを指定できます。
  • -location　　　　　　位置指定なし[default]
  • location location
    locationで指定する位置のデバイスのみに対応する

locationは現時点では以下のような文字列で表されます。

 location   意味 
 "USB-1"   "PlayStation 2"本体のUSBポート1 
 "USB-2"   "PlayStation 2"本体のUSBポート2 
 "USB-1,3"   "PlayStation 2"本体のUSBポート1に接続されたHubのポート3 

解説

locationの指定は、同じメーカーの同じ製品を同時に複数接続して使用したい場合に対応するためのもので、実用上これが必要になることはあまりないでしょう。

5. 5 dhcp

IPアドレスなどの設定をDHCPによって行うかどうかを指定するキーワードです。

書式

[-]dhcp

次のいずれかの指定ができます。
  • -dhcp　DHCPを使用しない[default]
  • dhcp　DHCPを使用する

5. 6 dhcp_host_name

DHCPサーバへ通知するホスト名を指定するキーワードです。

書式

次のいずれかの指定ができます。
  • -dhcp_host_name　　　　　　ホスト名を通知しない[default]
  • dhcp_host_name host_name　host_nameをホスト名として通知する

解説

DHCPサーバによっては、サーバ側から指定するホスト名をDHCPメッセージに含める必要があります。そのようなDHCPサーバのためのホスト名を指定するキーワードです。設定したホスト名はDHCP処理の際にのみ使用されます。

5. 7 dhcp_host_name_null_terminated

DHCPサーバへ通知するホスト名に末端のNUL文字を含めるかどうかを指定するキーワードです。

書式

次のいずれかの指定ができます。
  • -dhcp_host_namenull_terminated　ホスト名にNUL文字を含めない
  • dhcp_host_namenull_terminated　ホスト名にNUL文字を含める[default]

解説

本来、DHCPサーバへ通知するホスト名に末端のNUL文字を含める必要はありませんが、現在一般に使われているDHCPクライアントの多くがNUL文字を含めているため、それに合わせた設定をデフォルトとしてあります。特に問題が生じない限り、デフォルトのままにしてください。

5. 14 mtu

5. 19 chat_additional

5. 20 redial_count

解説

デフォルトのリダイヤル回数はNET_CNF中で設定されますが、ATTACH_CNFでは個々のインタフェースに依存する設定を行います。ATTACH_CNFでリダイヤル回数を設定した場合、NET_CNF中に設定があっても無視されます。

5. 21 redial_interval

書式

解説

モデムドライバが「通話中」を返した際にどれだけの時間を空けてからリダイヤルするかを秒単位で指定するキーワードです。指定しない場合のデフォルトは10秒です。
デフォルトのリダイヤル間隔はNET_CNF中で設定されますが、ATTACH_CNFでは個々のインタフェースに依存する設定を行います。ATTACH_CNFでリダイヤル間隔を設定した場合、NET_CNF中に設定があっても無視されます。

5. 22 outside_delay

外線発信遅延文字列を指定するキーワードです。PPP接続用のATTACH_CNF（type pppが指定されているATTACH_CNF）でのみ有効です。

書式

outside_delay "outside_delay"

解説

外線発信の際に、外線発信文字列と電話番号の間にはさむ外線発信遅延文字列を指定するキーワードです。モデムの機種と設定に依存しますが、多くのATコマンドモデムでは「,」で2秒遅延の意味になります。指定がない場合のデフォルトは""（空文字列）です。
デフォルトの外線発信遅延文字列はNET_CNF中で設定されますが、ATTACH_CNFでは個々のインタフェースに依存する設定を行います。ATTACH_CNFで外線発信遅延文字列を設定した場合、NET_CNF中に設定があっても無視されます。
なお、外線発信文字列が空文字列のときは外線発信遅延文字列は使用されません。

5. 23 dialing_type

解説

回線のタイプを次のいずれかで指定します。

 引数   意味

 any   上記以外の回線（デジタル等） 


デフォルトの回線タイプはNET_CNF中で設定されますが、ATTACH_CNFでは個々のインタフェースに依存する設定を行います。ATTACH_CNFで回線タイプを設定した場合、NET_CNF中に設定があっても無視されます。

5. 24 phone_number

発信先電話番号を指定するキーワードです。PPP接続用のATTACH_CNF（type pppが指定されているATTACH_CNF）でのみ有効です。

書式

phone_number "phone_number"
phone_number0 "phone_number"
phone_number1 "phone_number"
（省略）
phone_number9 "phone_number"

解説

発信先の電話番号を指定するキーワードです。
phone_number0からphone_number9は発信先の優先順位、つまりリダイヤルの順序を示します。まずphone_number0で指定された電話番号へ発信を試み、phone_number0が指定されていないか通話中であればphone_number1で指定された電話番号へ発信を試み、……というように順にリダイヤル処理が行われます（電話番号の指定がひとつしか存在しなければ、その同じ番号へ繰り返しリダイヤルが行われます）。
phone_numberとphone_number0は等価です。

ログイン手順を記述したスクリプト文字列を指定するキーワードです。PPP接続用のATTACH_CNF（type pppが指定されているATTACH_CNF）でのみ有効です。

5. 28 auth_name / auth_key / peer_name / peer_key

認証名および認証キーを指定するキーワードです。PPP接続用のATTACH_CNF（type pppが指定されているATTACH_CNF）でのみ有効です。

書式

auth_name "auth_name"
auth_key "auth_key"
peer_name "peer_name"
peer_key "peer_key"

解説

一般のISPから与えられるユーザ名とパスワードはauth_name / auth_keyに該当します。peer_name, peer_keyは接続先の認証名と認証キーです。いずれも、デフォルトの設定は存在しません。
各パラメータは、認証の方式に応じて次のように使用されます。

・PAP認証を相手側がこちらに要求する場合（相手側がサーバ）
 　auth_name   認証名として送信する 
 　auth_key   認証キーとして送信する 
・PAP認証をこちらが相手に要求する場合（相手側がサーバ）
 　peer_name   受信した認証名と照合する 
 　peer_key   受信した認証キーと照合する 
・CHAP認証を相手側がこちらに要求する場合（こちら側がサーバ）
 　auth_name   Responseで送信する認証名 
 　auth_key   Responseのダイジェスト化に使用する認証キー 
 　peer_name 
  		  Challengeで送られてくる相手側名と照合する 
 （peer_nameが"*"の場合は照合を行わない） 
・CHAP認証をこちらが相手に要求する場合（こちら側がサーバ）
 　peer_name   Responseの照合に使用する認証名 
 　peer_key   Responseの照合に使用する認証キー 
 　auth_name   Challengeで送信するこちらの認証名 

5. 29 want.auth

相手側に要求する認証方式を指定するキーワードです。PPP接続用のATTACH_CNF（type pppが指定されているATTACH_CNF）でのみ有効です。

書式

want.auth [any|pap|chap|pap/chap|chap/pap]

解説

相手に要求する認証方式を指定します。

 引数   認証方式 
 any   PAP,CHAPによる認証を要求しない [default] 
 pap   PAPによる認証のみを要求する 
 chap   CHAPによる認証のみを要求する 
 pap/chap   まずPAPによって認証を要求し、その結果接続を 
 拒否されたら、次にCHAPによって認証を要求する 
 chap/pap   まずCHAPによって認証を要求し、その結果接続を 
 拒否されたら、次にPAPによって認証を要求する 


CHAP認証で要求するアルゴリズムはMD5です。

5. 30 allow.auth

こちら側で許可する認証方式を指定するキーワードです。PPP接続用のATTACH_CNF（type pppが指定されているATTACH_CNF）でのみ有効です。

書式

allow.auth [any|pap|chap|pap/chap|chap/pap]

 引数   認証方式 
 any   PAP,CHAPによる認証を許可しない [default] 
 pap   PAPによる認証のみを許可する 
 chap   CHAPによる認証のみを許可する 
 pap/chap   PAPまたはCHAPによる認証を許可する 
 chap/pap   PAPまたはCHAPによる認証を許可する 


pap/chapとchap/papは同じ意味になります。CHAP認証で許可するアルゴリズムは、MD5 / MS(Version 1) / MS(Version 2)の３つです。

5. 31 lcp_timeout

address_nego, ip_address, ip_maskの指定とネゴシエーションの結果に応じて、IPアドレスは以下のように決まります。

 ネゴシエーション 
 要求 		  ネゴシエーション 
 結果 		  自分のIPアドレス 
 要求する 
 want.address_nego 		  Ack   want.ip_addressを使う 
 要求する 
 want.address_nego 		  Nak   与えられたIPアドレスとwant.ip 
 _maskとのANDを取り、それが

 要求しない 
 -want.address_nego 		  −   want.ip_addressを使う 


 ネゴシエーション 
 許可 		  ネゴシエーション 
 結果 		  相手のIPアドレス 
 許可する 
 allow.address_nego 		  Ack   allow.ip_addressを使う 
 許可する 
 allow.address_nego 		  Nak   送られてきたIPアドレスとallow.ip 
 _maskとのANDを取り、それが 
 allow.ip_addressとallow.ip_mask 
 とのANDと一致したらそのIPアド 
 レスを使う 
 許可しない 
 -allow.address_nego 		  −   allow.ip_addressを使う 


want.ip_address, allow.ip_addressのデフォルト値は"0.0.0.0"、want.ip_mask, allow.ip_maskのデフォルト値は"0.0.0.0"です。

 [-]want.dns1_nego    　[-]want.dns2_nego

5. 49 force_chap_type

許可するCHAP認証アルゴリズムを制限するキーワードです。PPP接続用のATTACH_CNF（type pppが指定されているATTACH_CNF）でのみ有効です。

書式

force_chap_type [no|md5|ms|ms-v1|ms-v2]

解説

CHAP認証を許可する場合にアルゴリズムを制限するキーワードです。引数の意味はそれぞれ次のとおりです。

 引数   意味 
 no   認証アルゴリズムを制限しない [default] 
 md5   MD5のみに制限する 
 ms   MS(Version1)のみに制限する 
 ms-v1   MS(Version1)のみに制限する 
 ms-v2   MS(Version2)のみに制限する 


指定しない場合は認証アルゴリズムを制限しません。
相手が要求した認証アルゴリズムで接続できない場合の回避策として特定のアルゴリズムを強制するために用意してあります。

5. 50 omit_empty_frame

空フレームの送信を制御するキーワードです。PPP接続用のATTACH_CNF（type pppが指定されているATTACH_CNF）でのみ有効です。

書式

[-]omit_empty_frame

解説

omit_empty_frameを指定すると、PPPフレームの直後の空フレーム（Flagのみのフレーム）を付加しないようになります。先頭に「-」を付けて-omit_empty_frameと指定すると、空フレームを付加するようになります。指定がない場合は付加するようになります。
受信時には空フレームの有無に関係なく受信可能です。

5. 51 log_flags

 flag   ログ表示内容 
 phase   PPPの状態 
 cp   LCP, IPCPの状態 
 auth   PAP, CHAPの状態 
 chat   チャット処理とその応答 
 private   チャットスクリプト中のプライベート情報 
 dll   DLLレベルでのデータのやり取り 
 dump   DLLデータのパケットダンプ 
 timer   タイマ関係の情報 
 event   PPPへのイベント 

　※それぞれ数値で指定することも可能です。

6 DIAL_CNFのキーワード

DIAL_CNFは、PPP接続の一般的なパラメータでモデムに依存するものを記述する設定ファイルです。

6. 1 dialing_type_string

ダイヤル文字列を定義するキーワードです。

書式

dialing_type_string "tone" "pulse" "any"

解説

チャットスクリプト内で使用する特殊文字列\Dの内容を定義します。ユーザが使用する回線タイプに応じて、ここで指定したtone / pulse / anyのいずれかが特殊文字列\Dに割り当てられます。
たとえば「dialing_type_string "ATDT" "ATDP" "ATD"」と指定しておくと、NET_CNF内でdialing_typeがtoneであれば、チャットスクリプト内の「\D」が「ATDT」に置き換えて実行されます。

6. 2 redial_string

通話中のときにモデムから返されるリザルト文字列を記述するキーワードです。

書式

redial_string "redial_string"

解説

通話中だった場合にモデムが返してくるリザルト文字列を記述するキーワードです。多くのATコマンドモデムでは「BUSY」です。

6. 3 chat_init

発信・着信手順の初期化部分を定義するキーワードです。

書式

chat_init "init_script"

解説

発信・着信双方で必要な初期化手順を指定したスクリプト文字列を指定します。この中には、redial_stringで指定した文字列を受信したらアボートする、という記述が含まれていなければなりません。
スクリプト文字列の仕様に関しては「チャットスクリプト仕様」を参照してください。

6. 4 chat_dial

発信手順を定義するキーワードです。

書式

chat_dial "dial_script"

解説

発信手順を指定したスクリプト文字列を指定します。スクリプト文字列の仕様に関しては「チャットスクリプト仕様」を参照してください。
NET_CNFまたはATTACH_CNFで-answer_modeが指定されていると、次のように発信用スクリプトが生成されます。
init_script + additional_script + dial_script

6. 5 chat_answer

着信手順を定義するキーワードです。

書式

chat_answer "answer_script"

解説

着信手順を指定したスクリプト文字列を指定します。スクリプト文字列の仕様に関しては「チャットスクリプト仕様」を参照してください。
NET_CNFまたはATTACH_CNFでanswer_modeが指定されていると、次のように着信用スクリプトが生成されます。
init_script + additional_script + "TIMEOUT answer_timeout" + dial_script

7 チャットスクリプトの仕様

チャットスクリプトは、PPP接続の際の、以下の5つの手順を指定するために使用するスクリプトです。
  • モデム初期化（chat_init）
    モデム接続の際のタイムアウト時間、アボート文字列、モデム初期化のスクリプト
  • モデム追加スクリプト（chat_additional）
    モデムに対してユーザ固有の設定を行うためのスクリプト
  • モデム発信スクリプト（chat_dial）
    発信処理のスクリプト。モデムドライバとともにモデムメーカーから供給される
  • モデム着信スクリプト（chat_answer）
    着信処理のスクリプト。モデムドライバとともにモデムメーカーから供給される
  • ログイン手続き（chat_login）
    PPPによる認証ではなく文字列のやり取りでのログイン手続きが必要な場合のスクリプト

7. 1 スクリプトの仕様

スクリプトはそれぞれひとつの文字列として記述されます。スクリプトをファイル内などで記述する際には一般的に「\」文字をエスケープして「\\」と記述する必要がありますので注意してください。

一般形式

一般的にスクリプトには、受信文字列と送信文字列を交互に、空白で区切って指定します。

recv1 send1 recv2 send2

この例では"recv1"を受信するまで待って"send1"を送り、次に"recv2"を受信するまで待って"send2"を送ります。

特殊文字列

スクリプト中で、以下に示す特殊文字はそれぞれ次のように置き換えて解釈されます。

 特殊文字   意味 
 \s   空白（SP:0x20） 
 \t   タブ（HT:0x09） 
 \r   復改（CR:0x0d） 
 \n   改行（LF:0x0a） 
 \U   認証名（auth_name） 
 \P   認証キー（auth_key） 
 \T   電話番号（phone_number） 
 \D   回線タイプ別ダイヤル文字列 
 ^X   CTRL-X（0x00-0x1f） 
 \c   文字列の最後に\rを付加しない（送信文字列専用） 
 \d   2秒待つ（delay）（送信文字列専用） 
 \p   0.25秒待つ（pause）（送信文字列専用） 
 \X   文字Xそのもの（特殊文字のエスケープ：送信文字列専用） 

空の文字列

何も待たずに送信する場合には、受信文字列として空の文字列を指定します。空の文字列は連続する2個のダブルクォートで記述します。

タイムアウト時間の指定

受信文字列を待つ上限の時間は次のように指定します。

TIMEOUT n	（nは10進数、単位は秒）

指定しない場合のデフォルト値は30秒です。受信文字列をタイムアウト時間内に受信できない場合は、PPPによる接続処理全体がエラーで終了します。

リトライの指定

受信文字列をタイムアウト時間内に受信できない場合のリトライ処理は次のように指定します。

recv1-send1-recv2 send2

"recv1"をタイムアウト時間内に受信すれば、そのまま"send2"を送信します。
"recv1"をタイムアウト時間内に受信しなかった場合には、"send1"を送信して"recv2"の受信を待ちます。"recv2"をタイムアウト時間内に受信すれば"send2"を送信し、そうでなければエラー終了します。

アボート文字列の指定

次の指定により、ある特定の文字列を受信した場合にエラー終了させることができます。

ABORT string

7. 2 チャットスクリプトの設定例

以下の設定例では、受信文字列と送信文字列とのペアを１行に１ペアずつ記述してあります。このようにすると確認が簡単になり、指定ミスも起こりにくくなります。
なお、NET設定ファイルはテキストで直書きできますが、説明のためC言語プログラム上での記述形式で示してあります。

chat_init =    		// < 初期化手続き >
     TIMEOUT 60		// タイムアウト時間を 60 秒に設定
     " ABORT NO\\sCARRIER"	// "NO CARRIER" 受信でエラー終了
     " ABORT ERROR"    	// "ERROR" 受信でエラー終了
     " ABORT BUSY"		// "BUSY" 受信でエラー終了
     " ABORT DELAYED"  	// "DELAYED" 受信でエラー終了
     " \"\" \\dA\\pT"  	// 遅延付きで "AT\r" を送信
     " OK \"\"";		// "OK" を待つ

chat_additional = 		// < 追加手続き >
     "";			//何もしないときは空文字列とする

chat_dial =    		// < 発信手続き >
     "\"\" \\D\\T"		// 発信する
     " CONNECT \\d\\c";	// "CONNECT"を待ち、さらに2秒待つ

chat_answer =  		// < 着信手続き >
     "RING ATA"		// "RING"を待って"ATA"を送る
     " CONNECT \\d\\c";	// "CONNECT"を待ち、さらに2秒待つ

chat_login =   		// < ログイン手続き>
     "TIMEOUT 10"		// タイムアウト時間を 10 秒に設定
     " ogin:--ogin: \\U"	// "ogin:" を再送付きで待って認証名を送信
     " ssword: \\P"    	// "ssword:" を待って認証キーを送信
     " PPP\\sstart";		// 最後に "PPP start" を待つ

7. 3 注意事項

  • 送信文字列中に「\c」を指定しない限り、送信文字列の最後には「\r」が自動的に付加して送信されます。受信文字列ではこのような自動的な付加処理は行われません。
  • 送信文字列と受信文字列の最大長は255バイトです。
    スクリプト全体の長さには（NET設定ファイルの仕様上では）特に制限はありません。
  • 受信文字列のバッファは送信文字列を送信する時点でクリアされます。
